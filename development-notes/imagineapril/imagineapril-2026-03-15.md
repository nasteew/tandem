## Code Review (Вариант Б, так как хотелось разобраться в подключении АИ, а этот пулл реквест уже был закрыт)
Я разбиралась в коде @anastasiashlyk. Это ИИ-сервис, с помощью которого пользователь может общаться с ИИ. Код содержит в себе и бекэнд, и фронтэнд. Весь процесс выглядит так: пользователь пишет запрос, который пересылается на сервер. Сервер пересылает его на API OpenRouter, получает ответ и пересылает его пользователю.

Бекэнд состоит из:
- `‎tandem/back-end/src/ai/ai.controller.ts‎` - контроллер, который получает запрос от пользователя, передаёт его сервису, а затем возвращает ответ пользователю
- `‎tandem/back-end/src/ai/ai.service.ts‎` - сервис, который отправляет запрос в OpenRouter и обрабытывает ответ
- `‎tandem/back-end/src/ai/ai.module.ts‎` - модуль, который объединяет контроллер и сервис.

Фронтэнд содержит:
- `tandem/front-end/src/pages/AgentPage/AgentPage.tsx‎` - компонент страницы агента
- `tandem/front-end/src/components/ui/Input/Input.tsx‎` - компонент формы ввода сообщения от пользователя
- `‎tandem/front-end/src/components/Message/message.tsx‎` - компонент ответа с печатающимся текстом

Подробнее разберу код в файле сервиса, потому что он содержит основную логику. Здесь проверяем API-ключ, отправляем запрос в OpenRouter fetch('https://openrouter.ai/api/v1/chat/completions'), получаем ответ от OpenRouter response.body в виде стрима (чанка) и отправляем этот чанк пользователю с помощью метода pipeStream.

Долго разбиралась с методом pipeStream. В итоге поняла так (надеюсь, что правильно 😀):

stream: ReadableStream<Uint8Array>  - кусочки данных (value) от OpenRouter (AI)
res: Response - ответ клиенту
getReader() - встроенный метод, который создаёт специальный объект для чтения потока по кусочкам
в цикле:
  reader.read() - читаем кусочек
  controller.enqueue(value) - отправляем клиенту (вот тут я зависла)
новый стрим new Response(newStream) - упаковываем новый стрим в ответ и pipeTo(res) -  отправляем клиенту

Сначала запуталась с двумя потоками: один приходит от OpenRouter, второй мы создаём, чтобы передать данные пользователю.

В следующем pull request тиммейт переписала этот кусок кода и стало понятнее: убрала `controller.enqueue(value)` и сразу писала ответ пользователю: `res.write(value)`. (Но в первом варианте мы могли бы ещё что-то сделать с этими данными при необходимости)

Для фронтэнда был создан компонент страницы, который содержал всю логику: стейт сообщений, логику отправки, разбор стрима, анимацию и вёрстку. В код ревью уже без меня предложили отличный план по рефакторингу — вынести логику в хук useChat, а анимацию и сообщения — в отдельные компоненты.
